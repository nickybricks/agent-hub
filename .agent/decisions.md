# Decisions

## In plain English (no jargon)
The Profile tab used to show two competing sections — a paragraph-style "Your persona" and a bullet-list "About you" — that could disagree because the chat updated only one of them. I merged them. The Profile tab now has a single "Your persona" section. When you correct a personal fact in chat, the assistant proposes a full new persona paragraph that folds in the correction; once you confirm, the Profile tab refreshes to match without a page reload.

## What changed
- `src/lib/chat-tools.ts` — removed the `remember_about_user` tool (spec + auto-run handler). Added a new `update_persona` mutating tool (spec, preview, executor) that replaces the canonical persona memory. Extended the `save_onboarding_answer` key enum description to include `name` / `bot_name` / `personal_context` so the onboarding warm-intro answers have a home.
- `src/lib/chat-agent.ts` — dropped the `soul` memory read + system-prompt injection block; the assistant now reads `user_profile` and injects it as the "what you know about this user" context block. Removed `hasSoul` from `onboardingState`'s return shape. Rewrote the STATE line to gate on the answered list (which now includes `name`). Updated `SYSTEM_PROMPT` to instruct the model to call `update_persona` with the full new text when correcting facts. Updated `ONBOARDING_SYSTEM_PROMPT` steps 2 and 5 to route warm-intro answers (name, bot name, optional personal context) through `save_onboarding_answer` instead of `remember_about_user`.
- `src/app/api/mail-analyzer/profile/route.ts` — removed the `soul` field from the response payload and the `soul` branch from the PUT handler. The route is now persona-only.
- `src/components/panes/ProfilePane.tsx` — deleted the "About you" section, its state (`soul`, `soulDraft`, `savingSoul`), the `saveSoul` handler, and the `soul` field from the `ProfileData` interface. The "Your persona" section is unchanged.

## Why these choices
- **Used the existing `user_profile` kind as the canonical store** rather than inventing a third kind. PATTERNS.md "Memory system" already lists `user_profile` as "synthesized persona", and `propose-structure.ts` and `greeting.ts` already read it — switching the chat agent to read it too means every consumer now reads the same canonical paragraph.
- **`update_persona` is a `mutate` tool** (requires Apply/Cancel confirmation) because the card called it a "mutate tool ... confirms". An auto-running variant would have meant fact-corrections silently overwrite the user-visible persona on every chat turn, which is the kind of behaviour PATTERNS.md "Anti-patterns" flags.
- **Picked "drop `remember_about_user`" over "trigger persona re-synth"** because the card explicitly called this out as the simpler option. Re-synth would have required running Sonnet on every personal fact and risked the persona drifting away from what the user just said — exactly the dual-source problem the card is trying to kill.
- **Route warm-intro answers through `save_onboarding_answer`** rather than auto-creating a partial `user_profile` pre-pipeline. The persona is supposed to be a synthesised paragraph that includes mailbox composition; producing one before the scan finished would be the wrong shape. `synthesizePersona` already reads every `user_pref` keyed `onboarding:*`, so `name` / `bot_name` / `personal_context` flow into the persona automatically when the pipeline runs.
- **Kept the existing `useDataBump` / `useRevalidate` plumbing** for the refresh-without-reload acceptance test. `ChatPanel` already bumps the global revision counter when any mutate tool finishes, and `ProfilePane` already calls `useRevalidate(active, load)`. No new sync wiring was needed.

## Judgment calls I made
- The chat system prompt now tells the model "call `update_persona` once with the new FULL persona text — fold the new fact into everything still true". This is a behavioural ask of Sonnet — it has to read the current persona out of the system prompt and rewrite it. The alternative phrasing would have been a more mechanical "append a sentence", but that re-introduces the drift problem (the persona becomes a junk drawer). I went with the rewrite framing because the card calls the tool a "replace" (`update_persona — replaces persona text`).
- The onboarding STATE line lost its dedicated "soul memory: saved/NOT yet" signal. I judged the answered list (now including `name`) sufficient to gate the warm-intro step, since the model can see the list and step 2's check is "if the `name` answer is not yet saved". The model also still sees the chat history, so even without the explicit gate it would not re-ask a name it already heard.
- I did NOT delete the `soul` enum literal from the `MemoryKind` union in `src/lib/analyzer-db.ts`. The card said "retire" / "drop the tool", not "drop the type". Leaving the literal in lets any pre-existing `soul` rows continue to type-check, and removing it would have rippled into the (legacy) SQLite helpers. Cheap insurance, zero behavioural cost.
- I extended the `save_onboarding_answer` key description string to enumerate the new keys. The schema itself is just `type: "string"`, so this is purely documentation for the model. I judged this clearer than letting the model guess what counts as a valid onboarding key.

## What I deliberately did NOT do
- **Did not delete `soul` rows from `agent_memory`.** Destructive, not authorized. Existing rows are now orphaned but harmless.
- **Did not remove the `"soul"` literal from the `MemoryKind` union.** Wider type refactor; not needed for the acceptance test.
- **Did not change `synthesizePersona` in `src/lib/onboarding.ts`.** It already reads every `onboarding:*` `user_pref`, which now naturally includes `name`, `bot_name`, and `personal_context`. No further folding-in needed.
- **Did not touch `src/agent/propose-structure.ts` or `src/lib/greeting.ts`** — both already read `user_profile`, so they automatically get the unified persona.
- **Did not touch `src/app/api/mail-analyzer/onboarding/persona/route.ts`** — it already writes to `user_profile`.
- **Did not update `docs/PATTERNS.md` or `docs/CHAT-FLOW.md`.** Both reference `soul` + `remember_about_user`. Engineer prompt's "Do not touch" list calls out the mandatory-reading docs, so I left them alone — flagged below as a follow-up.
- **Did not modify version, `package.json`, `.github/`, `tasks/`, `scripts/agent/`, or `playwright.config.ts`** per the hard rules.

## Risks and follow-ups
- **Doc drift.** `docs/PATTERNS.md` (Memory system section) still describes `soul` as a distinct kind and lists `remember_about_user` as a mutate tool. `docs/CHAT-FLOW.md` mentions `soul` in the memory-kinds list. Both now disagree with the code. Recommend a follow-up doc-only PR that's allowed to touch the mandatory-reading files.
- **No data migration of legacy `soul` rows.** Existing users who already accumulated personal facts in `soul` will silently lose them from the Profile UI on first reload after deploy. The facts remain in the database (`agent_memory` rows with `kind = 'soul'`) and could be one-shot folded into each user's `user_profile` via a script; I did not add that script because the card didn't authorize a migration. Worth deciding before deploy.
- **Onboarding flow is the most behaviour-sensitive change.** I verified types and a build pass, but the new "save name via `save_onboarding_answer key: name`" branch was not exercised end-to-end — only the model's prompt was changed. First post-deploy onboarding run is the real test.
- **The `update_persona` Apply card friction.** On every fact correction the user sees an "Apply / Cancel" card. That's the design the card asked for, but if users find it annoying for trivial corrections the right fix is an explicit "auto-confirm for persona edits" mechanism, not weakening the existing mutate contract.
- **Empty / `NaN` persona edge cases:** the executor throws if `content` is empty, matching how `previewMutation` handles it; the legacy PUT route also enforces non-empty now. Worth a manual sanity check that the chat UX surfaces that error nicely if the model ever calls with an empty body.
