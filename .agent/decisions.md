# Decisions

## In plain English (no jargon)
Sometimes during first-time setup, the chat got stuck on "Drafting your profile · Usually 5–30 seconds" forever, even though the profile had quietly finished on the server. I added a 5-second safety check: if the loading note hasn't gone away after 5 seconds, the chat quietly asks the server again. That second check is enough to rescue the screen and show the profile editor. A reviewer will notice one new small React effect in the chat panel — about ten lines — and nothing else.

## What changed
- `src/components/ChatPanel.tsx` — added a small one-shot watchdog `useEffect` right after the existing persona-draft fetch effect. When `pipeline?.phase === "persona_ready"` and `persona` is still null, it schedules a single 5-second `setTimeout`; on fire it increments `personaRetry`, which causes the existing fetch effect to re-run and POST again. Cleanup clears the timer when `persona` is set or the phase moves on.

## Why these choices
- **Defensive watchdog, separate effect, not a rewrite of the existing fetch logic.** The existing IIFE-based fetch effect handles definitive failures (POST returns no persona / throws) correctly — it schedules a 5s retry after the await settles. The bug is the case where the await *never* settles (response lost / Vercel timeout / network blip): no retry is scheduled, UI strands. Adding a watchdog that fires on a wall-clock timer, independent of the in-flight fetch, is the smallest change that closes that hole. Matches PATTERNS.md "Loading and progress UX → Defensive polling: if a 'ready' signal hasn't flipped after 5s past the expected completion, refetch once before giving up."
- **`personaRetry` deliberately left out of the deps array.** This means the watchdog fires at most once per entry into `persona_ready`, matching the card's "refetch once" wording. If it had been included, the watchdog would re-arm after every retry and the effect would poll forever, multiplying server load (since the first defensive refetch hits the route while the original synthesis is still in flight, which would spawn a duplicate synthesis).
- **Reused the existing `personaRetry` state** instead of inventing a new "watchdogFired" flag. The retry counter is exactly the signal we want — bumping it re-runs the fetch effect, which has all the right guards (`if (pipeline?.phase !== "persona_ready" || persona) return;`) and the right state side-effects (`setPersona` / `setPersonaEdit`). PATTERNS.md "Anti-patterns" warns against adding a second source of truth; one counter is enough.
- **Watchdog only inside `persona_ready`.** Other phases (scanning, classifying, proposing) have their own pipeline-polling tick every 4s, so they self-heal already. The strand only happens at `persona_ready` because that's the one phase whose progress signal depends on a *separate* POST, not just polling `/onboarding/pipeline`.

## Judgment calls I made
- **5-second threshold.** The existing loading copy already says "Usually 5–30 seconds", so a healthy fetch that's still in flight at the 5s mark *will* trigger a second POST. That's the intended behaviour for the strand case; for the happy path it just means one extra POST that finds the cached `onboarding_persona_draft` memory and returns immediately. If we wanted to be more conservative (say 15s) we'd lose responsiveness in the strand case. 5s matches the card and PATTERNS.md verbatim.
- **Did not abort the original in-flight POST.** No `AbortController` plumbing. The original POST's IIFE has a `cancelled` flag that flips when the fetch effect re-runs (which happens when the watchdog bumps `personaRetry`), so its post-await branch short-circuits before calling `setPersona`. If both syntheses end up running concurrently on the server, the route writes the memory once on whichever finishes first; the client gets whichever response races to land. No regression — just a small chance of one wasted Sonnet call in the strand case, which is the same case where the user was previously stranded with no UI feedback.
- **Did not unify the watchdog and the post-await retry into a single mechanism.** I considered restructuring the existing fetch effect to use `Promise.race` against a 5s timeout. It would be ~5 lines less code overall but the diff would touch the previously-working fetch logic, and CLAUDE.md "Surgical Changes" says every changed line should trace to the request. The additive watchdog is the smaller diff.
- **No new test.** `e2e/` is `landing.spec.ts` + `sign-in.spec.ts`; there is no onboarding fixture, the persona pipeline depends on Inngest + a real mailbox, and the strand condition itself is a race against a stuck fetch — hard to reproduce deterministically in Playwright without mocking the persona-draft route. Better tackled as a follow-up if onboarding e2e coverage gets added.

## What I deliberately did NOT do
- **Did not modify** `src/app/api/mail-analyzer/onboarding/persona-draft/route.ts`. The route is already idempotent (checks the cached `onboarding_persona_draft` memory before re-running Sonnet) — exactly what makes the defensive refetch cheap.
- **Did not modify** `src/app/api/mail-analyzer/onboarding/pipeline/route.ts`. Phase progression is correct; the bug was purely client-side.
- **Did not touch** the polling effect, the elapsed-timer effect, `loadThread`, `newChat`, `consume`, or any unrelated state.
- **Did not add** a global timeout/abort to the fetch itself. That would be a wider change touching the consumption pattern of every other route in this file.
- **Did not bump** the package version, per the engineer-task hard rules. The release flow handles that.
- **Did not append** to `docs/DECISIONS.md`. This is a bug-fix that fits the pattern already documented in `docs/PATTERNS.md` ("Defensive polling: ... refetch once") — no new architectural decision.

## Risks and follow-ups
- **Duplicate synthesis when the strand happens.** If the original POST is genuinely still in flight at the 5s mark, the watchdog's refetch will hit the route, find no cached memory yet, and start a *second* synthesizePersona call. Cost: ~5¢. Mitigation if it becomes a problem: add a server-side single-flight lock on the `onboarding_persona_draft` memory key. Out of scope for this card.
- **Watchdog fires once-per-entry**, so if both the original AND the defensive refetch fail (e.g. the Sonnet API is hard-down), the user still ends up stranded. The existing 5s post-await retry loop is supposed to cover the steady-state case where the route returns definitively but synthesis errored — but that path requires the await to settle. Worst case the user reloads. A more robust fix would be a wall-clock-based retry loop (every 5s while persona_ready + no persona), but the card asked for "once" so I stopped there.
- **Manual acceptance test still needs to happen.** From a clean account, trigger onboarding and confirm the draft appears within seconds of classify finishing without a reload. The repo's automated suite doesn't cover this path, and I can't simulate a stuck fetch in CI.
- **`useEffect` dep linter.** The new effect's deps are `[pipeline?.phase, persona]`, which are exactly the values it reads. No `eslint-disable` needed (and none added) — confirmed below in verify.
