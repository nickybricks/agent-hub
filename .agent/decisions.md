# Decisions

## In plain English (no jargon)
Before this change, correcting a personal fact in chat (e.g. "actually call me Mai") made the assistant pop up an "Apply / Cancel" card; the persona only updated after you clicked Apply, and the Profile tab then refreshed. After this change, the assistant just updates the persona right away — no card, no click — and the Profile tab refreshes on its own. Other things in chat that change real mailbox stuff (folders, rules, audit actions) still ask before doing anything.

## What changed
- `src/lib/chat-tools.ts` — flipped `update_persona`'s `kind` from `"mutate"` to `"read"`, dropped the "Requires user confirmation before it saves." sentence in its description (replaced with an "Auto-runs (no confirmation)" note explaining persona is low-stakes), moved its write handler from `executeMutation` into `runReadTool` (unchanged logic: insert new `user_profile` memory, supersede the prior one), and deleted the now-unused `update_persona` cases from `previewMutation` and `executeMutation`.
- `docs/DECISIONS.md` — appended an entry "2026-05-24 — Persona updates auto-apply (no confirm card)" documenting context, choice, why, what was rejected, and the explicit scope guard listing every other mutate tool that keeps its confirm gate.

## Why these choices
- **Used the existing `kind: "read"` shape rather than inventing a new `mutate_auto` literal.** `save_onboarding_answer` is already `kind: "read"` and writes a memory row — same pattern, already proven. PATTERNS.md's "Anti-patterns" calls out single-use abstractions, so adding a new `ToolKind` literal just for persona would have been speculative.
- **Moved the persona write logic into `runReadTool` rather than dispatching to `executeMutation` from there.** Keeping the logic in one place per kind matches the rest of the file: read tools live in `runReadTool`, mutating tools live in `executeMutation`. A cross-function dispatch would have left "what does this tool actually do?" answered in two functions.
- **Did NOT change `SYSTEM_PROMPT` or `ONBOARDING_SYSTEM_PROMPT`** in `src/lib/chat-agent.ts`. The model already gets told to call `update_persona` with the full new text; whether the runtime gates that with a confirm or auto-runs it is an implementation detail the model doesn't need to know. The bullet "Mutating tools require the user's explicit confirmation" stays accurate — it still applies to every tool still tagged `mutate`.
- **Did NOT touch the client (`ChatPanel.tsx`, `ProfilePane.tsx`, `DataSync.tsx`).** The existing read-tool branch already fires a `tool done` SSE event when a read tool finishes, which `ChatPanel.consume()` already pipes into `bump()`, which already triggers `ProfilePane.useRevalidate(active, load)` when the persona memory is re-read from `/api/mail-analyzer/profile`. The "Profile tab reflects the new persona without a page reload" acceptance criterion is satisfied by infrastructure that was already in place.

## Judgment calls I made
- **Replaced "Requires user confirmation before it saves." in the tool description with "Auto-runs (no confirmation) — persona is low-stakes and easily re-corrected in chat."** The original sentence was a contract the runtime no longer enforces; leaving it would mislead the model about how the tool behaves. The new sentence frames *why* — so if the model is reasoning about whether to call it on borderline corrections, the rationale ("low-stakes, easily re-corrected") nudges it toward calling rather than asking.
- **Did NOT add a "persona just updated — here's the new version" toast or visual flash** in the Profile tab when a chat-triggered update lands. The acceptance bar is "reflects the new persona without a page reload", which the existing `useRevalidate` mechanism already satisfies. A flash animation would be additive UI not in the card.
- **Did NOT add a backwards-compatibility shim for in-flight pending `update_persona` tool calls** that might exist at deploy time. If a user happens to have a pending `update_persona` row when the deploy lands, the `/api/mail-analyzer/chat/confirm` POST would fail with `unknown mutating tool: update_persona`. The window is tiny (pre-launch product, single-digit users, pending row only exists between the model's call and the user's click on the Apply card) and the user-visible failure is one chat error message that's recoverable by re-prompting. Per CLAUDE.md's "Don't use feature flags or backwards-compatibility shims when you can just change the code" — I chose to change the code cleanly.

## What I deliberately did NOT do
- **Did not change any other mutating tool's kind.** The card is persona-only. `rename_proposed_folder`, `set_proposed_folder_status`, `set_rule_status`, `update_rule_match`, `apply_rule`, `dismiss_audit_finding`, `set_audit_override`, `clear_pending_proposals`, `trigger_propose_structure`, `add_proposed_folder`, and `write_memory` all keep `kind: "mutate"` and their Apply/Cancel cards.
- **Did not edit `docs/PATTERNS.md`.** Its "Memory system" section still describes `update_persona` as a "Mutate tool" — that line is now stale, but the engineer prompt's "Do not touch" list calls out the mandatory-reading docs. Flagged as a follow-up below.
- **Did not edit `src/lib/chat-agent.ts`.** Re-read it to confirm; no prompt edit is required.
- **Did not touch the client refresh path** (`ChatPanel.tsx`, `DataSync.tsx`, `ProfilePane.tsx`). The existing `bump()` + `useRevalidate(active, load)` plumbing already covers the "refresh without reload" acceptance criterion for read tools.
- **Did not modify version numbers, `package.json`, `.github/`, `tasks/`, `scripts/agent/`, `playwright.config.ts`, or the mandatory-reading docs other than `DECISIONS.md` per the card's explicit instruction.**

## Risks and follow-ups
- **Doc drift in `docs/PATTERNS.md`.** The "Memory system" section's "Mutate tools" bullet still lists `update_persona`. Recommend a small doc-only follow-up PR that's allowed to touch mandatory-reading files.
- **In-flight pending `update_persona` tool calls at deploy time will 500.** See the judgment-call note above. Worth a quick `SELECT count(*) FROM tool_call WHERE tool_name = 'update_persona' AND status = 'pending'` before merging to be sure the count is zero (or near-zero).
- **The model now writes silently — verify the conversation still feels conversational.** Because the tool auto-runs and feeds a result back, the model gets one more loop iteration to produce a follow-up like "Got it — updated your profile." The risk is that the model sees the tool result and considers the turn complete without writing that follow-up. The system prompt says "Be concise and answer in the user's language", so it should write *something*, but the first real-user persona correction is the actual test.
- **No new test for the auto-apply path.** The repo has no unit test for chat-tools' kind dispatch and the e2e suite (`landing.spec.ts`, `sign-in.spec.ts`) doesn't exercise chat at all. Adding one is a separate body of work the card didn't authorize.
